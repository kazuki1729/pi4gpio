# ネットワーク制御方針（Network Policy）

`SESSION_HANDOFF.md` §4-3「ネットワーク越し制御の要否と、要る場合の認証方式」について決定した内容をまとめる。作業日: 2026-07-12。

## 1. 結論

- **リモート制御を許可する**
- 認証・到達性の制御は **Tailscale限定bind ＋ APIキー** の組み合わせで実現する
- **mTLSは当面見送り**
- 配布モデルは **「自分専用ソフトとして配布」**：各利用者が自分のRaspberry Pi 4に自分でインストールし、自分のTailscaleネットワーク・自分のAPIキーで完結させる。共有インスタンスは作らない

## 2. 背景：pigpioとの関係の整理

当初「pigpio＝センサーデータ取得のスレッディング化が目的」という理解があったが、実際にはpigpioは「GPIO/PWM/I2C/SPI/シリアル/1-Wireへのアクセスを複数プロセス（ローカル・リモート問わず）で共有する汎用ハードウェア制御デーモン」であり、**クライアント/サーバー方式でのリモート制御は後付けの機能ではなく設計の核**だった。ただし、その認証設計が甘く（pigpiodはデフォルトでTCP 8888を無認証で待受）、これが§2で挙げた不採用理由の一つになっている。

つまりpi4gpioがリモート制御を持つこと自体はpigpioの本質に忠実であり、問題は「認証をどう設計するか」に絞られる。

## 3. 選択肢の整理と選定理由

### ネットワーク層 vs アプリケーション層
- **ネットワーク層での委譲**（Tailscale、WireGuard）: デーモンは認証ロジックを持たず、認証済み経路からしか到達できないことだけを保証する
- **アプリケーション層での認証**（APIキー、mTLS）: デーモン自身が身元確認・鍵管理・失効の責任を持つ

両者は排他ではなく重ねて使える。

### Tailscale vs WireGuard
TailscaleはWireGuardの上に構築されたマネージドサービス（自動鍵交換・NAT越え・ACL管理付き）であり、独立した2択ではない。生のWireGuardを選ぶと鍵配布・ピア管理を自分で運用する手間が増えるだけで得るものが少ない。week09で既にTailscale運用の実績があるため、新しい技術スタックを増やさずTailscaleに統一する。

### mTLSを見送る理由
Tailscale自体が各ノードにWireGuard鍵ベースの強い身元証明とACLを与えている。mTLSを重ねる意味があるのは「自分のtailnet内の乗っ取られたノードすら信用しない」という高度な脅威モデルを想定する場合のみで、現時点では過剰。将来、具体的な脅威が明確になった時点で再検討する。

### APIキーを採用する理由
Tailscaleが与えるのは「到達性の認証」であり、「そのノード内でどのピン/バスに触れてよいか」という認可（authorization）の粒度までは提供しない。APIキーをピン/バス単位の権限スコープと紐付けることで、この粒度を埋める。

## 4. 配布モデルとその意味

`pip install`できるのは**クライアントライブラリ**のみ（`rpi-hw-lock`・`rpi-sensor-lib`と同じPyPI/Trusted Publishingパターンを流用可能）。**デーモン本体**はRust製のシステムサービスであり、以下の理由からpipでは配布しない。

- root権限相当のハードウェアアクセス（メモリマップドI/O、DMA制御）が必要で、pip installの過程で特権操作を行うのはサプライチェーンリスクとして避けるべき
- BCM2711（Pi 4固有）のレジスタマップに依存し、対象ハードウェアが限定的
- systemdサービスとしての永続起動、udevルールでのデバイス権限設定などOS統合作業が必要

デーモンは`.deb`パッケージまたはGitHub Releasesのバイナリ配布とし、systemdユニットも同梱する。

### 「自分専用ソフトとして配布」が意味すること

各利用者が自分のPi・自分のTailscaleネットワーク・自分のAPIキーで完結するため、**新規利用者が現れてもこちらが個別に何かする必要は発生しない**。これがこのモデルを選ぶ理由そのもの。

ただしこれを成立させるには、**キー発行やTailscale連携を利用者自身がセルフサービスでできる仕組みを事前に作り込んでおく**必要がある。

- デーモンにAPIキー生成コマンド（例: `pi4gpio keygen`）を持たせ、利用者が自分でその場で発行できるようにする
- Tailscale限定bindは設定ファイルで利用者自身が有効化できるオプションとして提供する
- あなた（メンテナ）が個々の利用者のtailnetに参加したり、キーを代理発行したりする設計にはしない

## 5. 実装時に必要な作業（宿題）

1. デーモンのbindオプション追加：Tailscaleインターフェースのみにリスンを制限する設定
2. APIキーの発行・保管・失効の仕組み（ハッシュ化して保存、キーごとの権限テーブル）
3. **クライアント識別子の統一**：ローカルソケットでは`SO_PEERCRED`（UID/PID）がクライアント識別子として使えるが、リモート経路では使えない。APIキー（またはTailscale IP）を識別子とし、`SESSION_HANDOFF.md` §3の「クライアントがロックを持ったままクラッシュした場合の自動解放」の設計をこれに合わせる
4. クライアントライブラリにAPIキーを渡すインターフェースを追加
5. 脅威モデリングを実施し`SECURITY.md`として明文化（APIキー漏洩時の被害範囲、Tailscaleノード乗っ取り時の被害範囲）

## 6. 利用者数と無関係に発生する、通常のOSS保守作業

新規利用者ごとの個別対応ではなく、公開ソフトウェアとして元々必要な作業。

- パッケージのリリース・更新（PyPI Trusted Publishing、`.deb`パッケージの更新）
- セキュリティ脆弱性が見つかった場合のパッチ対応
- セットアップ手順のドキュメント整備（一度整えれば繰り返し作業にはならない）
- ワイヤープロトコル・APIの後方互換性維持（破壊的変更時のマイグレーションガイド）
- GitHub Issueでの質問・不具合報告への対応（任意対応、他OSSと同じ立ち位置）

## 7. 段階的ロールアウト方針

いきなり公開するのではなく、まず自分のPiでTailscale＋APIキー構成を運用し、`rpi-sensor-lib`の実移行を通しで動かして実績を積んでから、その実績を持って公開する。
